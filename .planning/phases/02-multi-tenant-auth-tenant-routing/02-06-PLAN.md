    Create `lib/cache/tenant-cache.ts` (D-18, the ONLY sanctioned cache helper):
    - `export function tenantScopedCacheKey(tenantId: string, ...parts: (string|number)[]): string[]` -> returns `[tenantId, ...parts.map(String)]` so every key is tenant-prefixed.
    - `export function tenantCache<T>(tenantId: string, keyParts: (string|number)[], fn: () => Promise<T>, opts?: { tags?: string[]; revalidate?: number }): Promise<T>` -> a thin wrapper over Next's `unstable_cache` that:
       - uses `tenantScopedCacheKey(tenantId, ...keyParts)` as the cache key
       - prefixes every tag with the tenantId (e.g. `tags: (opts?.tags ?? []).map(t => \`${t}:${tenantId}\`)`) so a `revalidateTag` can never cross tenants
    - Provide a dev `Map` fallback ONLY if `unstable_cache` is unavailable in the runtime (research: a Map fallback is fine; almost nothing is cached today). Keep it minimal — the point is to establish the pattern before Phases 3-5.
    Add a JSDoc banner: "The only sanctioned cache helper. Every key + tag is tenantId-prefixed (TNT-07/D-18). Do not call unstable_cache directly elsewhere."

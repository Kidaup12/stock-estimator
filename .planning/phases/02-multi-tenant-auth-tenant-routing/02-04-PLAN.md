<objective>
Retrofit per-tenant timezone date bucketing (TNT-08, D-19) WITHOUT breaking Phase 1's determinism invariant. Add `lib/time/tenant-date.ts` (`date-fns-tz`), thread a tenant-local `runDateKey` (`YYYY-MM-DD` string) through the forecast simulator's seed AND the prediction's runDate bucket, and replace the UTC `new Date()` + `setUTCHours(0,0,0,0)` date math in the forecast/reorder paths with the tenant-tz helper.

Purpose: This is the landmine the research flagged (Pitfall 7). Phase 1's seed key `(productId, runDate)` flows through `rng.ts`'s `.toISOString().slice(0,10)` which is ALWAYS UTC. If `runDate` is now tenant-local but some paths still compute UTC, the same product on the same Nairobi day forecasts differently near UTC midnight (3am Nairobi). Computing `runDate` ONCE as `tenantDayKey(tz)` and passing the STRING into both the seed and the bucket is the fix.
Output: `lib/time/tenant-date.ts` + test, edited simulator + forecast-run route + run-forecasts script, extended determinism check.

Runs in Wave 3 (AFTER Plan 03) because it edits `app/api/forecast/run/route.ts` and `lib/forecast/simulate-layers.ts` — Plan 03 already touches the forecast-run route (findFirst swap), so serializing avoids a lost-write race.
</objective>

       ```typescript
       import { fromZonedTime, formatInTimeZone } from "date-fns-tz";
       /** YYYY-MM-DD in the tenant's tz — the determinism seed key component (D-19/D-06). */
       export function tenantDayKey(tz: string, when: Date = new Date()): string {
         return formatInTimeZone(when, tz, "yyyy-MM-dd");
       }
       /** UTC instant for tenant-local midnight "today" — for date-range filters. */
       export function tenantTodayUtc(tz: string, when: Date = new Date()): Date {
         const ymd = tenantDayKey(tz, when);
         return fromZonedTime(`${ymd}T00:00:00`, tz);
       }
       ```
